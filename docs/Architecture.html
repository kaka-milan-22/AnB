AnB · architecture

Key custody, split from the client.

Alice (the client your agents call) holds only ciphertext and runs redaction. Bob (the KMS daemon) holds the master key and serves encrypt/decrypt as an oracle over mutual TLS. The master key never lives on the client and never touches disk in plaintext.

Go · two static binaries AES-256-GCM values Argon2id key-wrap mutual TLS · private CA redaction engine

01The problem it solves

The predecessor stored a 256-bit master key in a plaintext file right next to the ciphertext — any process running as the same user could read it and decrypt everything. AnB removes the key from the client entirely and puts it behind an authenticated service.

Before

vault.key (plaintext) sits beside vault.json. One cat by a rogue dependency or a stolen backup = full compromise.

After

Alice keeps only ciphertext. The key lives in Bob, unlocked once by an operator, held in locked memory, and released to nobody — Bob decrypts for you.

03How the core operations flow

Every operation that touches the master key is a single authenticated round-trip to Bob. The key — and full-vault decryption power — stays on Bob.

set <key> — store a secret (human, TTY)
Alice reads the value (prompt / --from-env / --stdin) and sends the plaintext to Bob.
Bob checks authorization for the key, encrypts under the master key, returns the ciphertext.
Alice writes the ciphertext + metadata into vault.json. The plaintext is discarded.
read <file> — redact a file (safe, agent-callable)
Alice batches all vault ciphertexts and asks Bob to decryptMany in one request.
Bob authorizes each key, decrypts, returns the plaintext values.
Alice replaces known values + high-entropy tokens with <agent-vault:key> placeholders and prints the redacted file.
write <file> — restore placeholders (safe, agent-callable)
Alice extracts referenced placeholders, fetches just those ciphertexts, decrypts them via Bob.
Placeholders are substituted with real values; unknown keys fail closed with a clear error.
The restored file is written to disk.

04Features

Oracle, not key release

Bob encrypts/decrypts on request and returns only the result. The master key is never handed out, so a transient connection can't yield offline decryption of copied ciphertext.

Mutual TLS, private CA

Bob mints its own CA, server cert, and signs each client CSR. No public CA, no ACME. The client cert's CommonName is the caller's identity — Kubernetes-style.

Redaction engine

Faithful port of agent-vault: known-value masking plus high-entropy / pattern detection of unvaulted secrets, with sha256-keyed write-back restore.

Safe / sensitive split

Agents run safe commands without a TTY; set / get --reveal require an interactive terminal, so plaintext exfiltration is structurally blocked.

Per-identity authorization

Map each identity to allowed key prefixes. Every request is authorized and audit-logged.

Hardened key custody

Master key wrapped with Argon2id at rest, mlock'd in memory, core dumps disabled, PR_SET_DUMPABLE=0 on Linux, zeroized on idle TTL or shutdown.

05Components

crypto

AES-256-GCM (iv:tag:ct) + Argon2id key-wrap. The auditable core.

keystore

Bob's in-memory key holder: mlock, idle TTL, zeroize, process hardening.

ca

Private CA: generate, issue server/client certs, sign CSRs (ed25519).

mtls

tls.Config for both ends — require & verify client certs.

proto

Newline-JSON wire protocol: encrypt / decrypt / decryptMany / status.

server

Bob's oracle: identity from cert, authorize, audit, crypto.

client

Alice's mTLS RPC with typed errors (locked / unauthorized / unreachable).

redact

The redaction + restore engine (placeholders, entropy, patterns).

localvault

Alice's vault.json + connection config + cert paths.

06Trust boundary — read this

mTLS protects the wire and authenticates both ends. It does not protect the endpoints. Be honest about what this design does and doesn't buy:

Endpoints see plaintext

It lives in Bob's memory (it decrypts) and in Alice's process (it receives results). A compromised Alice gets the plaintext it asks for — unavoidable.

Client key = secret-zero

Whoever steals Alice's client private key can impersonate her to Bob. Keep it 0600; revoke and reissue if lost.

Bob is a central SPOF

It holds the KEK and sees all plaintext flowing through it. Harden and audit it as the crown jewel.

What you do gain over a plaintext key file

No usable key at rest — a stolen vault.json or backup is inert without the operator's master password. The key is never extractable, decryption is centrally authorized, audited, rate-limitable, and revocable per identity.

08Bootstrapping in three moves

# 1 · operator sets up Bob
bob ca init
bob init --host bob.internal,10.0.0.5     # mint server cert SANs
bob serve --addr :8443                      # unlock + listen (mTLS)

# 2 · Alice enrolls (CA handed over out of band)
alice enroll --identity alice-laptop --bob bob.internal:8443 \
             --server-name bob.internal --ca ./ca.crt
bob sign-csr alice-laptop.csr --out alice-laptop.crt   # operator reviews + signs
alice install-cert ./alice-laptop.crt
alice status                                # → Bob status: unlocked

# 3 · daily use
alice set stripe-key                        # human stores (Bob encrypts)
alice read config.yaml                       # agent reads — values redacted
echo 'token: <agent-vault:stripe-key>' | alice write config.yaml
